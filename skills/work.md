# 작업 - 코드 구현

## 전제 조건

- 이전 작업이 모두 마무리 된 상태.

## 실행 절차

- 준비 단계 -> 실행 단계로 나누어 진행.

### 준비 단계

1. `plan` 디렉토리에서 계획 확인. 현재 상태는 `plan/current.md` 확인.
2. 작업 전 브랜치 전환. `ai/{현재 작업}` 형태.
3. 작업 관련 android 가이드, 권장사항 확인.
4. 작업 계획 작성. 모든 작업이 하나의 commit에 들어가는게 아니라 작업 단위로 나누어 commit 진행되도록 설계. 각 commit은 빌드 가능해야 함.
5. 작업 계획을 핵심 요약하여 출력. 사용자 의사결정이 필요한 내용이 있다면 번호 있는 리스트로 정리하여 함께 출력.
6. 종료 후 사용자 답변 대기.
7. 작업 계획 수정.

### 실행 단계

1. 계획에 따라 commit 단위로 코드 구현.
2. 코드 수정 파일 `code-change.md`에서 실행 단계 진행.
3. `plan/current.md` 파일 수정.
4. 다음 commit 단위 구현 계속 실행. 모든 단위 완료 시 최종 완료 및 검증

## 제약 사항

- 작업 계획은 android 가이드와 kotlin 가이드를 준수해야 함.
	- android-cli 스킬 활용. android 가이드 확인 후 계획에 반영
	- kotlin Documentation(https://kotlinlang.org/docs/coding-conventions.html) 준수
- 코드, 주석, git commit message를 포함한 모든 텍스트는 간결하고 이해하기 쉽게 작성. 필요하지 않은 부차적인 내용과 미사여구 삭제.
- 각 진행 중인 단계 표시(예: `<WORK-준비-4>`, `<WORK-실행-6>`)
- 파일 수정 도구(`write_to_file`, `replace_file_content`, `multi_replace_file_content`, `Write`, `Edit` 등) 호출 전 변경 계획과 실제 코드(markdown)를 채팅 텍스트로 반드시 출력
- 각 파일마다 수정 계획과 코드 변경 사항 출력(다른 내용을 +- 형태로 알기 쉽게 표시) 후에 사용자 피드백(승인, 수정 지시)을 받아야 파일 수정 도구를 호출 가능
- 사용자의 수정 인가는 설명 받은 내용과 변경 사항에 한정되며, 다른 파일에 추가 수정이 필요한 경우 중단하고 사용자에 내용 출력.

## 예외(에러) 대응

- 계획에 따라 구현이 불가능한 경우, 중단하고 사용자에 내용 표시.
- 사용자 의사결정이 필요한 내용이 있으면, 중단하고 사용자 확인 후 계속 진행.

## 최종 완료 및 검증

1. 코드 수정 파일 `code-change.md`에서 검증 단계 재수행.
2. android, kotlin 가이드 준수 여부 최종 점검 및 요약 보고.
3. 검증 및 점검에 문제가 있으면 종료 및 해결 방안 출력.
4. `plan/current.md` 파일 작업 완료 상태 업데이트 및 commit. 메시지는 `docs: $현재_작업 구현 완료` 형태
5. git push
